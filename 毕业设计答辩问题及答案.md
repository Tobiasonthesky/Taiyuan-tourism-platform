# 基于Spring Boot的家乡文旅宣传平台 - 答辩问题及答案

## 一、项目概述类（高频问题）

### Q1: 请简单介绍一下你的项目？
**A:** 我的项目是基于Spring Boot和Vue.js的家乡文旅宣传平台，主要功能包括：
1. 景点、美食、文化、攻略等内容的展示与搜索
2. AI智能攻略生成，基于用户需求自动生成个性化攻略
3. 地图导航服务，支持地址解析、路线规划
4. 用户内容提交与审核机制
5. 个性化推荐系统
6. 酒店预订、评论收藏等互动功能

技术架构采用前后端分离，后端用Spring Boot + MyBatis Plus，前端用Vue.js + Element UI，集成了高德地图和DeepSeek AI服务。

### Q2: 你的项目有什么创新点？
**A:** 主要创新点有：
1. **AI智能攻略生成**：集成DeepSeek大语言模型，基于平台真实数据和用户需求（天数、预算、主题等）自动生成个性化旅游攻略
2. **智能地址解析**：用户输入地址自动解析为经纬度，审核时也会自动补充，提升数据录入效率
3. **完整业务闭环**：用户提交→管理员审核→内容发布→地图展示的完整流程
4. **多维度个性化推荐**：结合浏览历史、收藏记录、评分等实现混合推荐，数据不足时自动降级为热门推荐

### Q3: 为什么选择这个课题？
**A:** 选择这个课题主要基于以下考虑：
1. **实际需求**：家乡文旅资源丰富，但缺乏统一的数字化宣传平台
2. **技术实践**：可以综合运用所学的Spring Boot、Vue.js、数据库等技术
3. **社会价值**：为家乡文旅产业发展做贡献，提升家乡知名度
4. **创新空间**：可以结合AI、地图服务等新技术，有一定的创新性

## 二、技术架构类（高频问题）

### Q4: 请介绍一下你的系统架构？
**A:** 系统采用典型的前后端分离架构：
- **前端层**：Vue 2.6 + Element UI，负责页面展示和用户交互
- **后端层**：Spring Boot 2.7.6，采用MVC分层架构
  - Controller层：接收HTTP请求，参数校验
  - Service层：业务逻辑处理
  - Mapper层：数据访问（使用MyBatis Plus）
- **数据层**：MySQL 8.0存储业务数据，Redis可选作缓存
- **第三方服务**：高德地图API（地理编码、路线规划）、DeepSeek AI API（攻略生成、智能助手）

### Q5: 为什么选择前后端分离架构？
**A:** 选择前后端分离的原因：
1. **职责清晰**：前端专注UI交互，后端专注业务逻辑，便于并行开发
2. **技术选型灵活**：前后端可以独立选择技术栈
3. **便于维护**：前后端代码解耦，修改一方不影响另一方
4. **支持多端**：同一后端可以支持Web、App、小程序等多端
5. **性能优化**：前端可以独立做缓存、CDN等优化

### Q6: Spring Boot的优势是什么？为什么选择Spring Boot？
**A:** Spring Boot的优势：
1. **快速开发**：自动配置，减少繁琐的XML配置
2. **内嵌服务器**：内置Tomcat，无需单独部署
3. **起步依赖**：Starter依赖简化依赖管理
4. **生产就绪**：提供健康检查、监控等生产级特性
5. **生态丰富**：与Spring全家桶无缝集成，社区活跃

选择Spring Boot是因为它能快速搭建稳定的后端服务，大大提高开发效率。

## 三、核心功能实现类（高频问题，重点准备）

### Q7: 请详细介绍AI智能攻略生成的实现原理？（重点！）
**A:** AI攻略生成的实现流程（参考代码：`StrategyGenerateServiceImpl.java`）：

1. **数据准备**：从数据库获取热门景点（20条）、美食（15条）、酒店（10条）作为上下文数据
2. **提示词构建**：使用`PromptTemplateBuilder`构建结构化提示词，包括：
   - 系统角色定义（资深旅游规划专家）
   - 用户需求（天数、预算、主题、兴趣偏好）
   - 平台资源数据（景点、美食、酒店）
   - 输出格式要求（JSON格式）
3. **调用AI服务**：通过`AiServiceImpl`调用DeepSeek API，设置temperature=0.7，max_tokens=2000
4. **结果解析**：处理AI返回的内容，提取JSON数据，处理markdown代码块包裹的情况
5. **异常处理**：如果JSON解析失败，使用原始内容作为攻略正文
6. **返回结果**：构建Strategy对象，设置分类（根据天数）、状态（待审核）等

**关键代码位置**：
- 核心逻辑：`StrategyGenerateServiceImpl.generateStrategy()`
- AI服务调用：`AiServiceImpl.generateContent()`
- 提示词配置：`application.yml`中的`strategy.prompt`配置

### Q8: AI生成的攻略质量如何保证？
**A:** 保证攻略质量的措施：
1. **提示词工程**：精心设计提示词，明确要求AI基于平台真实数据生成
2. **输出格式约束**：要求AI返回JSON格式，便于结构化解析
3. **审核机制**：AI生成的攻略状态为"待审核"，需要管理员审核后才能发布
4. **用户编辑**：用户可以预览并编辑AI生成的内容后再保存
5. **降级策略**：如果AI服务不可用或返回异常，有相应的错误处理

### Q9: 地图服务是如何实现的？（重点！）
**A:** 地图服务实现（参考代码：`MapServiceImpl.java`）：

**主要功能：**
1. **统一地图数据接口**：整合景点、美食、酒店、文化的地理位置数据
   - 景点：直接经纬度字段
   - 美食：从推荐餐厅JSON字段中解析
   - 酒店：直接经纬度字段
   - 文化：活动地点经纬度

2. **地理编码（地址→经纬度）**：
   - 调用高德地图Geocoding API
   - 用户提交内容时前端自动解析
   - 管理员审核时如果地址存在但经纬度缺失，自动调用API补充

3. **路线规划**：
   - 支持驾车、步行、骑行三种模式
   - 调用高德地图Direction API
   - 实现降级机制：如果API调用失败，使用Haversine公式计算直线距离，根据模式估算时间

4. **地点搜索**：
   - 调用高德地图Place Search API
   - 支持城市限制（默认太原）
   - 坐标范围过滤，确保结果在目标城市内

**关键技术点：**
- 使用RestTemplate调用第三方API
- 配置化管理API密钥
- 完善的异常处理和降级策略

### Q10: 推荐算法是如何实现的？
**A:** 推荐系统实现（参考代码：`RecommendationServiceImpl.java`）：

采用**混合推荐策略**：

1. **热门推荐**（冷启动方案）：
   - 按浏览量、评分降序排序
   - 用于新用户或数据不足时

2. **个性化推荐**：
   - **基于浏览历史**：获取用户浏览过的内容分类，推荐同分类的其他内容，排除已浏览的
   - **基于收藏**：分析用户收藏的内容特征（分类、主题、星级等），推荐相似内容
   - **混合策略**：优先个性化推荐，数量不足时用热门推荐补充

3. **推荐流程示例（景点推荐）**：
   - 获取用户最近10条浏览历史
   - 提取浏览过的景点分类
   - 查询同分类、未浏览、状态正常的景点
   - 按评分、浏览量排序
   - 如果数量不足，补充热门景点

**特点：**
- 实现简单，易于理解和维护
- 有降级机制，确保推荐结果可用
- 支持多种内容类型（景点、美食、攻略、酒店等）

### Q11: 用户认证和权限控制是如何实现的？
**A:** 安全机制实现（参考代码：`SecurityConfig.java`）：

1. **认证方案**：JWT（JSON Web Token）+ Spring Security
   - 用户登录成功后生成JWT Token
   - Token包含用户ID、用户名、过期时间（24小时）
   - 前端在请求头中携带：`Authorization: Bearer {token}`

2. **密码加密**：使用BCrypt加密存储用户密码

3. **权限控制**：
   - 基于角色的访问控制（RBAC）：USER普通用户、ADMIN管理员
   - 接口级权限：使用`@PreAuthorize`注解，如`@PreAuthorize("hasRole('ADMIN')")`
   - 配置级权限：在`SecurityConfig`中配置公开接口、管理员接口、需要认证的接口

4. **JWT过滤器**：`JwtAuthenticationFilter`在每次请求时验证Token有效性

### Q12: 内容审核机制是如何实现的？
**A:** 内容审核流程（参考代码：`StrategyController.java`）：

1. **状态管理**：
   - 0：待审核
   - 1：已通过
   - 2：已拒绝

2. **用户提交**：
   - 普通用户提交的内容状态设为"待审核"
   - 管理员提交的内容直接"已通过"

3. **管理员审核**：
   - 查看待审核内容列表
   - 审核通过时：
     - 自动检查地址和经纬度
     - 如果地址存在但经纬度缺失，调用地图API自动解析
     - 状态改为"已通过"
   - 审核拒绝时：状态改为"已拒绝"

4. **内容展示**：只有状态为"已通过"的内容才会在前端展示

## 四、数据库与性能类

### Q13: 数据库是如何设计的？有哪些核心表？
**A:** 核心数据表：

1. **用户相关**：
   - `user`：用户信息（用户名、密码、角色、状态等）

2. **内容相关**：
   - `attraction`：景点
   - `food`：美食（含推荐餐厅JSON字段）
   - `culture`：文化（活动地点、时间）
   - `strategy`：攻略（AI生成、用户提交）
   - `hotel`：酒店（含房间信息）
   - `experience`：体验项目
   - 各内容表都有对应的图片表（如`attraction_image`）

3. **互动相关**：
   - `comment`：评论
   - `comment_reply`：评论回复
   - `favorite`：收藏
   - `browse_history`：浏览历史

4. **订单相关**：
   - `order`：订单
   - `order_item`：订单项

5. **系统管理**：
   - `announcement`：公告
   - `system_log`：系统日志
   - 分类表（景点分类、美食分类等）

**设计特点：**
- 使用逻辑删除（`deleted`字段）
- 状态字段统一设计（0待审核、1已通过等）
- 统计字段（浏览量、评论数、收藏数等）冗余存储，提高查询效率

### Q14: 如何解决数据库查询性能问题？
**A:** 性能优化措施：

1. **分页查询**：所有列表接口都使用MyBatis Plus的分页插件
2. **索引优化**：在常用查询字段上建索引（如状态、分类、排序字段）
3. **统计字段冗余**：浏览量、评论数等直接存在主表，避免关联查询统计
4. **连接池优化**：HikariCP连接池配置（最小空闲5，最大20）
5. **Redis缓存**（可选）：可以缓存热门数据、用户会话等
6. **查询优化**：使用LambdaQueryWrapper构建高效查询，避免N+1问题

### Q16: 开发过程中遇到的最大困难是什么？如何解决的？
**A:** 最大的困难是**AI攻略生成的稳定性和质量控制**：

**遇到的问题：**
1. AI返回内容格式不固定，有时不按要求返回JSON
2. AI API调用失败或超时
3. AI生成的内容可能包含平台不存在的信息

**解决方案：**
1. **格式容错**：在解析AI返回内容时，处理markdown代码块包裹的情况，增加异常捕获，解析失败时使用原始内容
2. **降级机制**：配置AI功能开关，API调用失败时有友好提示；使用配置化的超时设置（max_tokens=2000）
3. **提示词约束**：在提示词中明确要求AI"严格限定于提供的资源列表，不得自行添加或虚构内容"
4. **审核机制**：AI生成的内容必须经过管理员审核才能发布

## 六、其他可能的问题

### Q17: 为什么选择DeepSeek而不是其他AI模型？
**A:** 选择DeepSeek的原因：
1. **性价比高**：相比GPT-4等价格更低
2. **中文能力强**：对中文理解和生成效果好，适合中文旅游攻略
3. **API稳定**：提供稳定的REST API接口
4. **配置灵活**：支持temperature、max_tokens等参数调节

### Q18: 地图服务为什么选择高德而不是百度？
**A:** 选择高德地图的原因：
1. **国内数据更准确**：高德在国内地图数据方面更精准
2. **API文档清晰**：文档完善，易于集成
3. **免费额度充足**：对个人开发者友好，免费调用次数足够
4. **项目兼容**：代码中预留了地图类型配置（type: amap/baidu），方便切换

### Q19: 如何保证系统的安全性？
**A:** 安全措施：
1. **认证授权**：JWT + Spring Security，BCrypt密码加密
2. **接口鉴权**：基于角色的访问控制
3. **SQL注入防护**：使用MyBatis Plus的参数化查询
4. **XSS防护**：前端对用户输入进行转义
5. **CORS配置**：合理配置跨域访问
6. **操作日志**：记录关键操作，便于审计
7. **API密钥保护**：配置文件中管理密钥，不上传到代码仓库

### Q20: 项目中使用了哪些设计模式？
**A:** 主要使用的设计模式：
1. **MVC模式**：经典的Model-View-Controller分层架构
2. **策略模式**：推荐算法中，根据不同数据情况选择不同推荐策略
3. **模板方法模式**：提示词构建使用模板，结合用户需求动态填充
4. **单例模式**：Spring管理的Bean默认是单例
5. **工厂模式**：MyBatis Plus的MapperFactoryBean
