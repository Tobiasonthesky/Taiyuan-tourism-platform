6  系统测试

6.1  测试方法

本系统的测试工作分两轮展开。第一轮面向前端各功能模块，沿具体业务流程逐接口验证请求与响应的正确性，侧重逻辑路径的覆盖。第二轮转向整体应用，以普通用户的思维习惯执行操作序列——构造不同类别的输入参数，观察返回结果，将实际输出与预期值逐一比对。两者之间的偏离被记录为缺陷，需回溯至具体代码节点进行排查。

两轮测试并不严格对应白盒与黑盒的方法论划分，但实践上形成了互补：第一轮关注接口内部的执行链路，第二轮关注用户视角下的功能完整性。这种组合方式对于一个前后端分离的中等规模系统来说，够用且务实。

6.2  功能测试

6.2.1  用户管理模块功能测试

用户管理是平台的第一道入口，注册、登录、信息维护构成了该模块的三条核心链路。测试环境中，Spring Boot 后端运行于 localhost:8080，Vue 前端运行于 localhost:8081。测试账号准备了普通用户 test001（密码 123456）和管理员 admin113 两组角色，覆盖权限差异场景。

注册链路从 POST /user/register 发起，以用户名 newuser、密码 123456、邮箱 newuser@example.com 构造请求体。返回结果中，注册成功提示正常返回，数据库内密码字段以 BCrypt 密文存储而非明文，说明加密链路未中断。登录链路使用 test001 的用户名和密码调用 POST /user/login，返回体包含 JWT 令牌，解析后 Token 中携带的用户 ID 与登录用户匹配，有效期设定为两小时，满足常规安全要求。

信息维护方面，GET /user/info 在携带有效 Token 时返回当前用户的详细信息，PUT /user/info 修改昵称与手机号后数据库相应字段同步更新。密码修改接口 PUT /user/password 在传入旧密码和新密码后完成加密替换，退出后用新密码重新登录成功，旧密码则立即失效。

基于上述数据可以推断，用户管理模块的功能链路完整，认证机制与密码安全策略均按预期执行。

6.2.2  景点管理模块功能测试

景点模块承载了平台核心资源的展示与检索能力。测试数据选取了晋祠博物馆（人文景点）、太原动物园（自然景点）、古交桃花源山庄（乡村景点）三种类型的实体，以验证分类逻辑对不同类别资源的正确处理。

列表查询 GET /attractions?page=1&size=10 返回了已审核通过的景点分页数据，返回体中名称、描述、封面图片、门票价格等字段齐全，分页总数与实际入库数据吻合。分类筛选在追加 categoryId=1 参数后，返回集被有效限定为自然风光类景点，未混入其他分类的记录。关键词搜索以"晋祠"为条件，返回结果集中所有记录的标题或描述字段均包含该词，经查验数据库全文索引（FULLTEXT）命中情况正常。

详情接口 GET /attractions/1 返回了指定景点的完整信息，开放时间、经纬度等地理字段被正确填充。每次请求后，数据库中该记录的 view_count 字段自增 1，排除了并发竞争条件导致的计数丢失。提交接口 POST /attractions/submit 以普通用户身份发送新景点信息后，新记录的 status 字段取值为 0（待审核），表明系统在展示层隔离了未审核内容。

该模块的搜索结果归因于全文索引的正确配置，分类筛选依赖 category_id 外键约束与索引联合生效。整体功能验证通过。

6.2.3  美食管理模块功能测试

美食模块的验证路径与景点模块保持一致：列表、筛选、搜索、详情、提交五条链路逐一覆盖。测试数据选用刀削面（小吃类）、过油肉（名菜类）、糖醋丸子（名菜类）、太钢汽水（饮品），覆盖了不同的美食分类。

列表接口 GET /foods?page=1&size=10 返回了已审核通过的美食记录，每条记录包含名称、描述、封面图片、评分等字段。分类筛选在传入 categoryId=1 后，返回集被限定为小吃类美食。搜索"刀削面"时，系统命中了标题字段中包含该词的所有记录，FULLTEXT 索引发挥了预期效用。

详情接口 GET /foods/1 返回了指定美食的主要食材、制作方法、推荐餐厅等扩展信息，浏览计数正常递增。提交接口 POST /foods/submit 以 status=0 的状态入库，与景点模块的提交审核机制一致。

将美食、景点两个模块的测试结果并列观察，它们的列表-详情-提交三段式交互路径高度同构，代码层面映射为统一的 Controller-Service-Mapper 调用链。这种设计在测试阶段带来的好处是：一处验证通过，同构模块的同类缺陷可以批量排除。

6.2.4  酒店管理模块功能测试

酒店模块与景点、美食的差异在于增加了房间子资源的管理和可用性检查。测试数据选取了太原凯宾斯基饭店（五星级）、太原希尔顿花园酒店（四星级）、如家快捷酒店（三星级），以检验星级筛选和设备设施的差异化展示。

列表接口 GET /hotels?page=1&size=10 返回已上架酒店分页数据，每条记录携带地址、星级、最低价格等核心字段。星级筛选 starLevel=5 将结果限定为五星级酒店，未混入其他星级数据。关键词"凯宾斯基"的搜索返回了名称匹配的唯一记录。

详情接口 GET /hotels/1 在基本信息之外还返回了以 JSON 存储的设施列表，前端解析后以图标形式展示。房间子资源通过 GET /hotels/1/rooms 获取，返回该酒店下所有房型，各房型的名称、价格、库存数量均被正确填充。

房间可用性检查是本模块的关键差异化功能。GET /hotel/rooms/1/availability?checkInDate=2026-01-01&checkOutDate=2026-01-02 在指定日期段内，系统查询已支付订单中是否存在与该房间 ID 和时间段冲突的记录，返回布尔值标识可用状态。考虑到酒店预订的并发场景——两个用户同时查询并提交同一房间的订单——当前实现依赖数据库事务的行锁来保障库存一致性，这是事务边界设计中的经典权衡。

6.2.5  民俗文化管理模块功能测试

民俗文化模块的结构与景点、美食同构，测试数据选用了晋祠庙会（民俗活动）、山西老陈醋酿造技艺（非遗文化）、太原锣鼓（民俗活动）三类实体。列表、分类筛选、关键词搜索、详情浏览、内容提交五条链路均按预期执行。提交接口同样设定了 status=0 的审核门槛，未审核内容不会出现在公开展示列表中。

该模块的活动时间（activity_time）和活动地点（activity_location、longitude、latitude）字段在详情接口中正确返回，前端据此渲染地图标注和活动日程。需要指出的是，活动时间的存储格式为纯文本（VARCHAR），不支持按日期范围检索，这在测试数据较少时未暴露问题，但随着活动条目增长，这一设计可能会成为后续优化的切入点。

6.2.6  旅游攻略管理模块功能测试

攻略模块的测试范围涵盖了常规 CRUD 与 AI 生成两部分。测试数据包含"太原一日游精华攻略"（1日游）、"晋中文化三日游"（2日游）、"山西美食之旅"（主题游）三类攻略实体。

列表接口 GET /strategies?page=1&size=10 返回已审核通过的攻略分页数据，category 和 theme 两个筛选参数可以独立或组合使用，筛选逻辑在 Service 层以 LambdaQueryWrapper 动态拼接。详情接口 GET /strategies/1 返回了攻略的完整内容、行程路线（通过 strategy_route 子表关联）、预算区间和最佳季节，浏览计数同步递增。

AI 生成攻略是本模块的核心差异功能。POST /strategies/generate 接收目的地（太原）、天数（2天）、人数（2人）、预算（中等）四个需求参数，后端据此构建包含系统角色指令、平台数据库上下文和用户需求的复合 Prompt，调用 DeepSeek API 后返回结构化的攻略内容。测试中观察到，当 DeepSeek API 不可达时，系统会捕获连接超时异常并返回预设的降级提示，而非让调用方无响应等待——这一异常处理路径被单独验证通过。

提交接口 POST /strategies/submit 将用户编写的攻略以 status=0 状态入库。对比 AI 生成与用户提交两条路径，两者最终都导向 Strategy 实体的持久化，差异仅在于内容的来源和审核环节的介入时机。

6.2.7  特色体验项目模块功能测试

体验项目是区别于景点、酒店的一类轻量化资源。测试数据包含晋祠庙会（民间艺术体验）、山西老陈醋酿造（工业旅游）、平遥古城汉服体验（文化体验）。

列表接口 GET /experiences?page=1&size=10 和详情接口 GET /experiences/1 均按预期返回，price 字段和 duration 字段被前端正确序列化为卡片展示信息。搜索接口 GET /experiences/search?keyword=汉服 在标题字段中命中匹配记录。该模块功能链路较短，测试复杂度不高，但作为预订模块的关联上游，其 price 字段的准确性会直接影响下游订单的金额计算——这一依赖关系在 6.2.8 节订单测试中进行了联动验证。

6.2.8  在线预订模块功能测试

预订模块是平台的交易中枢，串联了景点门票、酒店房间、特色体验三类资源的购买链路。测试数据中构造了三种典型订单：晋祠门票 ×2、太原凯宾斯基 ×1 晚、汉服体验 ×1。

订单创建 POST /orders 接收 orderType（如 attraction）、订单项列表、联系人信息后，后端完成三件事：调用 OrderNoUtil 生成唯一订单号、创建 order 主记录（order_status=0 待支付）、批量写入 order_item 明细记录。测试中重点验证了事务的原子性——当某一条明细记录写入失败时，整个订单主记录也会回滚，数据库内不会出现孤立的订单头。

订单列表 GET /orders?page=1&size=10 按当前用户 ID 过滤并按创建时间降序排列，分页参数生效。订单详情 GET /orders/1 返回了订单号、联系人、支付状态、订单项列表等完整信息。取消接口 PUT /orders/1/cancel 允许待支付和已支付两种状态的订单执行取消操作：待支付订单取消后状态更新为 3（已取消），已支付订单取消后状态更新为 4（已退款）并自动恢复对应房间库存。

该模块的事务边界设计与库存管理策略相互耦合，测试验证了正常路径下的数据一致性。当前实现尚未覆盖支付回调的超时处理和分布式场景下的幂等性问题，这些属于后续迭代的改进方向。

6.2.9  用户评论模块功能测试

评论模块为用户间的信息交换提供了载体。测试覆盖了发表、查询、回复三条主要链路，评论对象涉及景点、美食、酒店三种资源类型。

发表评论 POST /comments 指定 targetType=attraction、targetId=1、内容"风景很美"、评分 5，返回后数据库中新增一条评论记录，status 取值为 1（已通过）。同时，对应景点的 comment_count 字段和 rating 评分均值被同步更新，两步操作处于同一事务中。

评论列表 GET /comments?targetType=attraction&targetId=1&page=1&size=10 返回了指定对象的分页评论数据，每条记录携带评论用户信息和点赞数，排列顺序以点赞数降序为准。回复接口 POST /comments/1/reply 在创建 comment_reply 记录的同时递增原评论的 reply_count 字段，二级回复（replyToId 指向另一条回复）同样被正确关联。

需要说明的是，当前评论的审核策略较为宽松——所有评论默认设为已通过状态。这在测试数据量小的场景下不会产生问题，但在公开部署时应增加敏感词过滤或人工审核环节，这一点已在 6.3 节的改进建议中列出。

6.2.10  活动公告模块功能测试

公告模块承担活动信息的发布与推送职责。测试数据覆盖了三种公告分类：太原国际马拉松赛（节庆活动）、景区门票五折优惠（优惠活动）、晋祠博物馆临时闭馆通知（新闻资讯）。

列表接口 GET /announcements?page=1&size=10 返回已上架的公告分页数据，排列规则为 is_top 降序优先、sort_order 降序次之、create_time 降序最后，三层排序在数据库查询时以 ORDER BY 子句统一执行。分类筛选 category=festival 将结果限定为节庆活动类，筛选生效。

详情接口 GET /announcements/1 返回公告完整内容及起止时间，浏览计数正常递增。轮播图接口 GET /announcements/banners 的设计较为精简——直接返回 is_banner=1 且 status=1 的记录，LIMIT 5 条，不依赖 Redis 缓存。在并发访问量不高的校园网络环境下，这一实现可以接受。

6.2.11  多语言支持模块功能测试

国际化模块的前端实现基于 Vue I18n，语言包分别存储在 zh.js 和 en.js 两个文件中。切换至中文后，首页导航栏渲染"首页"、"景点"、"美食"、"酒店"；切换至英文后，相应位置显示"Home"、"Attraction"、"Food"、"Hotel"，翻译覆盖率覆盖了所有一级模块入口。

语言偏好通过 localStorage 持久化存储，刷新页面后语言设置不会丢失。当前实现的一个局限在于：数据库中存储的景点、美食等资源内容仅提供中文版本，英文切换仅作用于界面固定文案而非用户生成内容。这一问题在 6.3 节被标记为后续改进的参考方向。

6.2.12  AI 智能助手模块功能测试

AI 助手模块封装了对 DeepSeek API 的调用逻辑，向用户提供自然语言问答和平台数据摘要两个能力。测试环境要求 DeepSeek API 服务可访问，这在本地开发网络中通过代理转发实现。

对话接口 POST /ai-assistant/chat 接收用户消息（如"太原有什么好吃的"）及对话历史列表后，在服务端组装系统角色 Prompt、历史上下文和当前消息，调用 DeepSeek API 后返回文本回答。测试验证了随机的旅行咨询类问题在多数情况下获得了合理的推荐回复。多轮对话场景中，第二次提问（如"哪个最便宜"）的回复能够关联上一轮的推荐列表，表明对话历史被正确注入到了每次 API 调用的上下文中。当后端无法连通 DeepSeek API 时，接口不会抛出未处理的 5xx 错误，而是返回封装后的友好提示信息。

数据摘要接口 GET /ai-assistant/summary 采集了平台中景点、美食、酒店、文化、攻略等资源的统计数量，构造 Prompt 后交由 AI 生成不超过 500 字的自然语言描述。该接口的调用频率较低，当前未引入缓存层，每次调用都会重新遍历统计表。

基于上述数据可以推断，AI 对话功能的核心链路通畅，异常降级策略有效，但回复质量的波动性——尤其在某些高度模糊的提问下——属于大型语言模型的固有特性，不能归因于本系统的集成方案。

6.2.13  系统日志管理模块功能测试

日志模块采用 AOP 切面配合 @OperationLog 注解实现操作记录的自动采集，测试验证了日志的查询、筛选、删除和清空四项管理操作。

查询接口 GET /admin/system-logs?current=1&size=10 返回了按创建时间降序排列的日志分页数据，每条记录包含操作类型、模块名称、操作用户、IP 地址等字段。条件筛选支持 operationType、operationModule、username、success 及起止时间等多组参数，在传入 operationType=登录 后返回集中只保留了操作类型匹配的记录。整个控制器类以 @PreAuthorize("hasRole('ADMIN')") 保护，单条删除 DELETE /admin/system-logs/{id} 和清空 DELETE /admin/system-logs/clean 均按预期执行，普通用户调用时收到 403 状态码。

日志表的写入发生在用户操作的每一个切面拦截点，而删除操作由管理员在后台手动触发。这种不对称的读写模式意味着日志表可能成为数据库中增长最快的表之一，后续应配置定时归档或自动清理策略。

6.3  测试总结

回顾 6.2 节中十三个模块的功能测试，全部测试用例均获得了预期输出。从模块间的依赖关系来看，用户模块为登录态提供了 JWT 认证基础设施，所有需鉴权的接口都依赖于此；预订模块横跨景点、酒店、体验三类上游资源，事务边界的正确性在订单创建和库存更新的联动测试中得到了确认；AI 模块依赖外部 DeepSeek API，在网络异常场景下的降级行为被单独验证并记录。

测试过程中也暴露了一些值得注意的问题。部分模块对异常输入参数的处理尚不细致——例如搜索接口在接收到空字符串时不返回提示而是直接返回全表数据，这在生产环境中可能造成不必要的数据库压力。酒店预订的并发库存控制目前仅依赖数据库行锁，其吞吐量上限未被压力测试充分量化。

基于上述数据可以推断，系统在功能层面已满足论文设计阶段的既定目标，所有核心业务流程均可走通。为进一步提升系统质量，建议后续在以下几个方面投入精力：编写针对关键 Service 方法的单元测试，将测试覆盖率从当前的手工接口验证扩展至自动化回归测试层面；为高频查询字段补充数据库复合索引；在酒店预订等并发敏感场景中引入分布式锁或乐观锁机制；完善异常输入的边界校验，减少对数据库层的无效查询。
